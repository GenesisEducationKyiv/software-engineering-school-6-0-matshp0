# 0003. Kysely для запитів у рантаймі, Prisma лише для міграцій

- Статус: accepted
- Дата: 2026-05-10

## Контекст і проблема

Потрібно обрати спосіб роботи з PostgreSQL: як писати запити в рантаймі та як
вести міграції схеми. Запитів небагато, але вони мають бути типобезпечними і
без оверхеду на трансляцію об'єктної моделі. Окремо потрібні зрілі міграції
з історією та командами `migrate dev`, `migrate deploy`, `migrate status`.

## Драйвери рішення

- сильна типізація запитів без класів-моделей і без рантайм-проксі
- мінімальний рантайм-оверхед і прозорий згенерований SQL
- зрілий інструмент міграцій з підтримкою `dev`/`deploy`/`status`
- мінімум абстракцій між кодом і SQL

## Розглянуті варіанти

- **Тільки Prisma** — і для запитів, і для міграцій. Простіше, але рантайм
  Prisma-Client тягне окремий двигун, генерує власні типи моделей, ховає
  фактичний SQL та обмежує складніші запити.
- **Тільки Kysely** — типобезпечні запити, але немає вбудованого інструменту
  міграцій; довелося б писати міграції руками або брати щось на кшталт
  `kysely-migrator`.
- **Drizzle ORM** — типи + міграції в одному пакеті. Зріла альтернатива,
  але міграційний воркфлоу менш звичний за Prisma; на момент рішення менша
  спільнота.
- **TypeORM / Sequelize** — клас-орієнтовані ORM, надлишкові для нашої
  моделі; декоратори, активні рекорди, прихований SQL.
- **Гібрид Kysely + Prisma migrate** — Kysely в рантаймі, Prisma виключно як
  CLI для міграцій. Схема описується в `schema.prisma`, типи Kysely
  генеруються окремо з фактичної схеми БД.

## Рішення

Обрано **гібрид: Kysely для запитів + Prisma тільки для міграцій**. Жодних
імпортів `@prisma/client` у рантаймі немає — Prisma запускається лише через
npm-скрипти. Запити пишуться через Kysely з типами, що відображають реальні
таблиці.

## Наслідки

**Позитивні**

- запити повністю типобезпечні: помилка в назві колонки ловиться компілятором
- SQL прозорий і налагоджуваний — можна логувати, EXPLAIN, оптимізувати
- немає Prisma Query Engine у рантайм-контейнері — менше залежностей у проді
- міграції — стандартний Prisma-воркфлоу, який добре документований

**Негативні / компроміси**

- два інструменти замість одного — більше когнітивного навантаження
- типи Kysely треба тримати синхронізованими зі схемою; зміна моделі вимагає
  міграції + регенерації типів
- дублікація опису схеми: `schema.prisma` як джерело істини для міграцій,
  згенеровані `database/types.ts` для запитів
