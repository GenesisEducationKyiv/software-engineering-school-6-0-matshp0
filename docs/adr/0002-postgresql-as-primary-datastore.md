# 0002. PostgreSQL як основне сховище

- Статус: accepted
- Дата: 2026-05-10

## Контекст і проблема

Сервіс зберігає дві сутності: репозиторії, за якими треба стежити, і підписки
користувачів. Дані реляційні: одна підписка належить одному репозиторію, на
один репозиторій підписано багато email-ів. Потрібні унікальні обмеження —
наприклад, `(email, repository_id)` — і генерація UUID-токенів для confirm /
unsub. Очікуване навантаження невелике: десятки тисяч підписок, сотні
репозиторіїв, RPS на API одиничні.

## Драйвери рішення

- природна реляційна модель з зовнішніми ключами та унікальними обмеженнями
- транзакційні гарантії при підтвердженні / відписці
- генерація UUID на стороні БД (`gen_random_uuid()`)
- зрілість, передбачуваність, локальний запуск через Docker
- безкоштовно і без постачальника

## Розглянуті варіанти

- **PostgreSQL 16** — реляційна БД, повний SQL, UUID, partial indexes,
  транзакції, JSONB якщо знадобиться.
- **MySQL / MariaDB** — порівнянний варіант, але слабша підтримка UUID без
  розширень, менш зручні `RETURNING` і часткові індекси.
- **SQLite** — надлегкий, але один файл-БД ускладнює багатопроцесний
  деплой і кластеризацію в майбутньому.
- **MongoDB** — модель не реляційна, відсутність FK і транзакцій між
  колекціями штучно ускладнила б `subscription -> repository`.

## Рішення

Обрано **PostgreSQL 16**. Дані реляційні, обсяги невеликі, інші варіанти або
менш зручні для UUID/обмежень, або не дають відповідних гарантій.

## Наслідки

**Позитивні**

- унікальність `(email, repository_id)` гарантується БД, а не кодом
- `confirm_token` і `unsub_token` — UUID, генеруються в DEFAULT, отже не
  залежать від мови застосунку
- можна додати часткові індекси, наприклад на `status = 'confirmed'`,
  якщо вибірки стануть «гарячими»
- легкий локальний стек: контейнер PostgreSQL у `docker-compose`

**Негативні / компроміси**

- порівняно з SQLite — потрібен окремий процес і конфігурація
