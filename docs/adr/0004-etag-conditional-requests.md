# 0004. ETag-кондиційні запити для сканера

- Статус: accepted
- Дата: 2026-05-10

## Контекст і проблема

Сканер детектує нові релізи через polling GitHub REST API. Polling масштабується
лінійно: N репозиторіїв × частота тіків = запитів на годину. На безкоштовному
GitHub-токені ліміт — 5000 запитів на годину. Без оптимізації 1000 репозиторіїв з
інтервалом 5 хв з'їдять ліміт повністю (~12 000/h).

GitHub підтримує conditional requests: якщо клієнт надсилає
`If-None-Match: <etag>` і нічого не змінилося, відповідь — `304 Not Modified`.
Такі відповіді **не списуються** з основного rate limit.

## Драйвери рішення

- мінімізувати споживання rate limit, особливо коли в репозиторіях нічого
  не змінюється (типовий випадок)
- мати чесну реакцію на `429 Too Many Requests`, а не сліпо ретраїти
- зберігати простоту коду — без зовнішнього кешу

## Розглянуті варіанти

- **Без кешування**: кожен тік — `GET /releases/latest` з повною відповіддю.
  Повністю споживає ліміт, неприйнятно.
- **`If-Modified-Since` за `last_checked_at`** — підтримується GitHub, але
  ETag точніший, бо не залежить від годинників і часових зон.
- **ETag, що зберігається в колонці `repositories.etag`** — ETag живе разом
  із записом репо, оновлюється атомарно з `last_seen_tag` і
  `last_checked_at`.

## Рішення

Обрано **ETag, збережений у колонці `repositories.etag`**. Сканер на кожен
тік:

1. читає репозиторії в порядку `last_checked_at ASC`;
2. для кожного робить запит з `If-None-Match: <etag>`;
3. на `304` оновлює лише `last_checked_at`;
4. на `200` порівнює `tag_name` з `last_seen_tag`, і якщо новий — нотифікує
   підписників, оновлює `last_seen_tag` і `etag`;
5. на `429` зчитує `Retry-After` / `x-ratelimit-reset`, ставить
   `suspendedUntil` і пропускає всі подальші запити до моменту скидання.

## Наслідки

**Позитивні**

- більшість тіків коштують `0` запитів на rate limit, бо більшість
  репозиторіїв не релізяться щохвилини
- стан polling-у атомарний з основними даними репо — без окремого кешу
- `429`-логіка централізована у `scanner.service.ts` і вимикає весь сканер,
  а не лише поточний цикл

**Негативні / компроміси**

- якщо GitHub змінить семантику ETag-ів (наприклад, через зміни в API),
  логіка детекції може стати менш ефективною; це ризик, але історично
  стабільний контракт
- ETag треба охайно оновлювати і на гілці `200`, і на гілці `304`, інакше
  деградує до запитів без кешу — є тести / лог, що це покривають
- після граничних випадків `etag`
  залишається `null`; перший справжній реліз створить його
