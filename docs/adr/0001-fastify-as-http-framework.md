# 0001. Fastify як HTTP-фреймворк

- Статус: accepted
- Дата: 2026-05-10

## Контекст і проблема

Сервіс надає невелике REST API: підписка, підтвердження, відписка, отримання
підписок, `/metrics`. Окрім самого HTTP, нам потрібен механізм модульного
підключення компонентів — config, БД, GitHub-клієнт, mail, repositories,
services, scanner — з контрольованим порядком завантаження та явними
залежностями. Треба обрати фреймворк, який не нав'язує важку структуру, але
дає вбудовану валідацію схем і прийнятну продуктивність.

## Драйвери рішення

- вбудована валідація request/response через JSON Schema
- система плагінів із декларативними залежностями та хуками життєвого циклу
- TypeScript-friendly без додаткових прошарків
- легка інтеграція Prometheus-метрик і структурованого логування
- достатня продуктивність для polling-сервісу з помірним RPS

## Розглянуті варіанти

- **Express 5** — фактичний стандарт, але без вбудованої валідації, без
  типобезпечного DI, плагінна модель робиться вручну через middleware.
- **NestJS** — повноцінний DI-каркас, але надлишковий для CRUD з 4 ендпоінтами;
  тягне декоратори, модулі, провайдерів, що збільшує вхідний поріг для
  навчального проєкту.
- **Hono / Elysia** — швидкі та сучасні, але менша екосистема плагінів
  (`fastify-plugin`, autoload, metrics, swagger), менш зріла інтеграція з
  Node-екосистемою.
- **Fastify 5** — плагінна архітектура, JSON Schema з коробки, `fastify-plugin`
  + `@fastify/autoload`, pino-логер, прозоро TS-friendly.

## Рішення

Обрано **Fastify 5**. Він покриває всі драйвери одночасно, а ключова перевага —
плагінна модель з декларацією залежностей: ми отримуємо передбачуваний порядок
завантаження `config → infrastructure → repositories → services → modules`
без власного DI-контейнера.

## Наслідки

**Позитивні**

- валідація вхідних даних описується разом зі схемою маршруту, помилки
  валідації стандартизовані
- порядок ініціалізації перевіряється фреймворком — якщо плагін оголошує
  залежність, відсутність декоратора виявиться при старті, а не в рантаймі
- pino дає JSON-логи без додаткових пакунків
- екосистема: `@fastify/autoload`, `@fastify/metrics`, `@fastify/swagger`

**Негативні / компроміси**

- плагінна модель з `fastify-plugin` / encapsulation — додатковий концепт,
  який треба засвоїти; помилки в інкапсуляції плагіна можуть давати
  «декоратор не видно» без очевидної причини
- інтерфейс `FastifyInstance` розширюється через `declare module`, що менш
  очевидне за класичний DI
- менша спільнота, ніж в Express; не всі сторонні бібліотеки мають готові
  Fastify-плагіни
